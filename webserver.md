2. HTTP Fundamentals

HTTP stands for:

Hypertext Transfer Protocol

It is the protocol used for communication between clients and servers on the web.

HTTP follows a request-response model.

HTTP Request

Example request:

GET /users HTTP/1.1
Host: example.com
User-Agent: Chrome
Accept: application/json

Important parts:

Method

The action the client wants to perform.

Common methods:

Method	Purpose
GET	Retrieve data
POST	Create new resource
PUT	Update resource
PATCH	Partial update
DELETE	Delete resource
HTTP Response

Example:

HTTP/1.1 200 OK
Content-Type: application/json

{
 "name": "John"
}

Components:

Status Code

Indicates result of request.

Common codes:

Code	Meaning
200	Success
201	Created
301	Redirect
400	Bad request
401	Unauthorized
403	Forbidden
404	Not found
500	Server error
3. URL vs URI

Students often confuse these.

Example:

https://example.com/users?id=5

Breakdown:

Protocol: https
Domain: example.com
Path: /users
Query parameter: id=5
URI

Uniform Resource Identifier

Identifies a resource.

URL

Uniform Resource Locator

A type of URI that specifies location of the resource.

8. Reverse Proxy

A reverse proxy sits between clients and backend servers.

Architecture:

Client
  |
  v
Reverse Proxy (Nginx)
  |
  v
Backend Servers

Responsibilities:

request routing

SSL termination

load balancing

caching

security filtering

Example Nginx configuration:

location /api {
    proxy_pass http://backend:3000;
}

Meaning:

Requests to /api will be forwarded to the backend service.

LAB — Reverse Proxy + Load Balancing

Run three web servers using Docker.

docker run -d -p 8001:80 nginx
docker run -d -p 8002:80 nginx
docker run -d -p 8003:80 nginx

Each server returns a different page.

Then configure Nginx as a load balancer.

Example config:

upstream backend {
    server localhost:8001;
    server localhost:8002;
    server localhost:8003;
}

server {
    listen 80;

    location / {
        proxy_pass http://backend;
    }
}

Now requests are distributed across servers.

Test using:

curl localhost

Students will see responses from different servers.

Session 2 — Load Balancing, Caching, Security & API Gateways
1. Load Balancing

Load balancing distributes incoming traffic across multiple servers.

Architecture:

        Load Balancer
       /      |      \
      v       v       v
   Server1 Server2 Server3

Benefits:

higher availability

better performance

fault tolerance

Load Balancing Algorithms
Round Robin

Requests are distributed sequentially.

Request 1 -> Server1
Request 2 -> Server2
Request 3 -> Server3
Least Connections

Traffic goes to the server with the fewest active connections.

Better for APIs.

IP Hash

Requests from the same IP go to the same server.

Useful for session persistence.

2. Caching

Problem:

User -> Backend -> Database

If thousands of users request the same data, the database becomes overloaded.

Solution: caching.

Architecture:

User
 |
 v
Nginx Cache
 |
 v
Backend

If cached data exists, Nginx returns it without contacting backend.

Example configuration concept:

proxy_cache_path /tmp/nginx_cache keys_zone=my_cache:10m;

Benefits:

faster responses

reduced backend load

improved scalability

3. Forward Proxy

Forward proxy sits between clients and the internet.

Architecture:

Client -> Proxy -> Internet

Uses:

corporate networks

anonymity

filtering

VPN systems

Difference:

Reverse Proxy	Forward Proxy
Protects servers	Protects clients
Clients don't see backend	Internet doesn't see client
